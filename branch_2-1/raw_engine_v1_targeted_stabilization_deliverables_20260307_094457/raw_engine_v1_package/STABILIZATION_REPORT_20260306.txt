2. targeted planner generator family
- split_ready
  - targetOcc starts sole
  - discover_split_pair_from_support() is guaranteed
  - planner executes an actual split
- split_with_boundary_artifact
  - targetOcc starts sole and split-ready
  - split result always contains one boundary-only child
  - afterSplit queues INTEGRATE_CHILD and planner executes integrate
- split_with_keepOcc_sibling
  - setup builds a real split result with a keepOcc sibling child
  - planner starts from the post-split state plus the afterSplit queue
  - afterSplit queues JOIN_PAIR and planner executes join
- split_with_join_and_integrate
  - setup builds a real split result with both a keepOcc sibling child and a boundary-only child
  - planner starts from the post-split state plus the afterSplit queue
  - planner executes integrate, join, then ensure_sole(targetOcc)
- planner_mixed_targeted
  - deterministic round-robin over the four targeted families, rotated by seed
  - avoids isolate-only dominance by construction

- afterSplit remains the only place that constructs JOIN_PAIR / INTEGRATE_CHILD

- src/ primitive implementation was not modified in this phase

11. remaining risks
- random and weighted planner traces are still isolate-heavy; targeted coverage closes the gap for split/join/integrate, but organic discovery in random families remains weak
- join coverage still depends on post-split targeted setup because the default planner entry condition for split is `targetOcc is sole`; this keeps join candidate discovery highly state-shape-sensitive
- stats `total_iterations` counts scenario iterations, while primitive hit counters count planner primitive executions; that is intentional for seed replay, but the two numbers should not be compared directly
- existing compile warnings in tests/ around unqualified `move` and aggregate initialization remain; they predate this phase except for the targeted planner path that was already cleaned up where needed
