3. policy graduation gate definition
- production default remains `exact_shadow`
- `exact_full` remains tests/debug/nightly compare-only
- new CLI / checkpoint knobs:
  - `--target-compared-states`
  - `--target-eligible-states`
  - `--stop-when-gate-passes`
  - `--max-partial-runs`
- threshold table in `policy_graduation_threshold_for_family(...)`:
  - `split_tie_ready = 16 / 16 / 16`
  - `split_tie_organic_symmetric = 32 / 32 / 32`
  - `planner_tie_mixed_organic_compare_ready = 32 / 32 / 32`
  - `automorphism_probe_large = 32 / 32 / 32`
- effective target rules:
  - `effectiveComparedTarget = max(configuredCompared, --target-compared-states)`
  - `effectiveEligibleTarget = max(configuredEligible, --target-eligible-states)`
  - `effectiveCompletedTarget = max(configuredCompleted, --target-compared-states, --target-eligible-states)`
- status rules in `evaluate_policy_graduation_gate(...)`:
  - `FAIL`
    - any semantic disagreement
    - any fallback
    - any semantic shift
  - `INSUFFICIENT_EVIDENCE`
    - compared / eligible / completed counts below effective targets
  - `PASS`
    - targets met
    - representative shifts absent
  - `SOFT_PASS`
    - targets met
    - representative shifts present
    - semantic shift still zero
- aggregate outputs now always emit:
  - `gate_status`
  - `stop_reason`
  - `target_compared_states`
  - `target_eligible_states`
  - `compare_completed_state_count`
  - per-family `run_gate=...`

10. remaining risks
- representativeness remains indirect:
  - direct `planner_tie_mixed_organic` compare eligibility is still absent in the sampled/rerun campaigns
  - the acceptance argument depends on the lineage audit proving that compare-ready contributes only the missing compare structure
- compare runtime is still high:
  - especially under ASan and in semantic-shift / lineage compare paths
  - nightly operations should continue to rely on checkpoint/resume rather than assuming one-shot completion
- exact-full pair-evaluation cache hit ratio remains `0.0` in the final compare campaigns
- stop-reason nuance:
  - when a family reaches gate in the same run that also requested `--stop-after-checkpoint`, `gate_passed` wins over `stop_after_checkpoint`
  - this is acceptable, but it matters when writing staged automation around resume tests
