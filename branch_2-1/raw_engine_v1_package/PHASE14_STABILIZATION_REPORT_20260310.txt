2. Checkpoint / resume design
- Added campaign checkpoint CLI:
  - `--checkpoint-dir`
  - `--checkpoint-every`
  - `--resume-from`
  - `--max-wall-seconds`
  - `--stop-after-checkpoint`
- Each chunk checkpoint persists:
  - run/seed/iter window
  - stats json path
  - summary path
  - compare/fallback counters
  - campaign manifest snapshot
- Resume logic now:
  - loads existing chunk manifests
  - skips already completed `(runIndex, seedIndex, seed, iterStart, iterCount)` chunks
  - reconstructs run aggregates from chunk summaries
  - rewrites per-run and aggregate outputs after each completed chunk
- tests-only correctness fix applied in this phase:
  - chunk summary format was expanded to include primitive hits, histograms, oracle mismatch map, and per-seed summaries
  - summary loader now restores those fields
  - `merge_fuzz_stats_into()` now merges `seedSummaries` instead of dropping them
- Resume evidence:
  - `artifacts/checkpoints/resume_smoke_run2/chunks`: 2 chunks
  - `artifacts/checkpoints/split_tie_organic_test_run4/chunks`: 2 chunks

10. Remaining risks
- `planner_tie_mixed_organic` still under-produces split-choice compare states in sampled campaigns, so exact_full adequacy evidence there remains weaker than for `split_tie_organic_symmetric`.
- Even with checkpoint/resume and caching, exact_full compare remains operationally expensive on symmetry-heavy families.
- Current fallback metric is still a combined field; in this phase all observed fallback counts were zero, so interpretability was not impacted, but a future non-zero campaign would benefit from separate policy-vs-compare fallback counters.
- Aggregate campaign summaries remain intentionally compact; the full-fidelity checkpoint data now lives in chunk summary files, which resume uses for reconstruction.
- Clean ASan full-matrix isolation on fresh build dirs works, but the reused `/tmp/cpp-checker-phase14-asan` tree can still be polluted by stale detached runners if old sessions are left behind.
